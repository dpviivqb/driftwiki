# "文档漂移检测"商业可行性报告

> 诚实评估,不画饼。结论先行:**值得做,但别重投入——开源引流 + 云增值,目标被收购或小而美。**

## 1. 一句话机会

把 deepwiki-skill 这类"生成器"当**编译器**,做它的 **linter / CI**——检测已有文档与代码的漂移。生成赛道已经免费化(repwiki / deepwiki-skill 开源),但"检测漂移"**还没有人做**,是一个窄但真实的缺口。

## 2. 市场与竞品

| 产品 | 定位 | 定价 | 增量更新 | 漂移检测 | 开源 |
|------|------|------|---------|---------|------|
| repowiki | 全量 wiki CLI | 免费(自付 API) | ❌ | ❌ | ✅ |
| deepwiki-skill | wiki skill + 增量 | 免费(自付订阅) | ✅ | ❌ | ✅ |
| DeepWiki(Cognition) | 公开 repo wiki | 免费(商业随 Devin) | 部分 | ❌ | ❌ |
| 通义灵码 Repo Wiki | 企业 IDE 集成 | 企业版付费(Beta 有免费额度) | ✅ | ❌ | ❌ |
| Mintlify / GitBook | 文档平台 | 商业(询价) | — | ❌ | ❌ |

> 价格以官网为准;上表为定性对比。

**价位带**(开发者工具锚点):个人 $10–19/月(Copilot 档),团队/企业 $25–50+/seat/月(Snyk / Sourcegraph 档)。

## 3. 差异化定位

漂移检测 = **文档版 CI**(docs-as-tests)。类比定位:
- Snyk 扫依赖漏洞 → 漂移工具扫"文档漏洞"
- Dependabot 依赖过时提醒 → 文档过时提醒
- type-checker 只校验不生成 → 漂移检测只校验不生成

**真实痛点(成立)**:
- API 契约漂移(文档端点 ≠ 代码端点)→ 集成方投诉
- SDK 文档与实现不符 → 开发者踩坑
- 金融/医疗合规审计要求文档与代码一致
- 新人按过时文档操作出错

**护城河判断(诚实):浅。** 漂移检测核心算法(提取事实 + 对账)不难,deepwiki-skill / repowiki 完全可以加这功能。先发窗口约 **6–12 个月**。真正壁垒要靠:规则库积累(多框架多语言提取器)、CI 深度集成、企业 on-prem。

## 4. 目标客户

- **最匹配**:API 平台公司(对外文档准确性 = 信誉)、SDK 提供商、活跃开源项目维护者(文档 issue 多)
- **次匹配**:中大型研发团队(文档量大、腐化快)、金融/医疗合规行业
- **不匹配**:个人开发者/小项目(文档少,不值得付费)

## 5. MVP 与定价

**MVP(最小可行)**:
- 后端事实提取器(先做 FastAPI + Express 两个高价值框架)
- 对账引擎 + Markdown/JSON 报告
- GitHub App:PR 评论标注漂移
- 先支持 REST 端点 + 配置常量(最典型、最痛的漂移)

**定价模型(开源核心 + 云增值,推荐)**:
- **Free**:开源 CLI,本地跑漂移检测
- **Team $15/seat/月**:GitHub App 自动 PR 评论、团队 dashboard、多框架
- **Enterprise $49+/seat/月**:on-prem 部署、SSO、合规报告、SLA

对标 Copilot $19 / Snyk $25 的中间档。

## 6. 护城河与风险

**护城河构建**:
1. **规则库**:积累多框架多语言提取器(FastAPI / Django / Spring / Express / NestJS / gRPC / protobuf / OpenAPI…)—— 越多越难复制
2. **CI 深度**:GitHub/GitLab 集成、PR inline 评论、漂移趋势图
3. **企业**:on-prem + 合规审计报告(金融医疗刚需)
4. **飞轮**:检测 → 自动触发 deepwiki-skill 修复(捆绑生成 + 检测)

**风险**:
- ❌ **最大风险**:大厂(Copilot / 通义灵码 / Sourcegraph)直接加这功能,小厂被碾
- ❌ 开源竞品(deepwiki-skill)加漂移检测,免费化
- ❌ 获客贵:开发者工具获客成本高,LTV 要撑住
- ❌ 技术易复制:核心不难,先发窗口窄

## 7. 诚实结论

**值得做,但别重投入。**

- 窗口期存在(没人做漂移检测)但窄(6–12 个月,大厂/开源会跟进)
- **推荐路径:开源引流 + 云增值**,别做重型企业销售
  - 开源核心(漂移检测 CLI)免费,攒用户 + 规则库
  - 云增值(GitHub App 自动化、dashboard)收费
  - 退出:被大厂收购(最可能),或维持小而美
- **不推荐**:直接做企业 on-prem 销售(获客贵、周期长,个人/小团队撑不住)
- **关键:速度。** 先开源,把 FastAPI / Express / Spring 三个框架的提取器做扎实,占心智

**如果只是自用/本项目**:不用做商业版。按《漂移检测设计》写个脚本挂 CI 就够——那才是本项目该走的路。

## 8. POC 验证(已跑通,2026-06-13)

[`scripts/drift_check.py`](../scripts/drift_check.py)(纯 stdlib)已在本项目跑通,差异化**已实证**:

- **端点漂移检测**:`ast` 提取代码 **18 个**端点 vs `REPOWIKI.md` 的 **15 个**,精准抓出 **3 个漏记**,每条带 `file:line` 溯源:
  - `GET /extension-auth/diagnostics` ← `extension_auth.py:56`
  - `GET /orders/list` ← `orders.py:33`
  - `POST /auth/access-token` ← `auth.py:77`
- **cite 行号校验**:校验 `07_backend-api.md` 的 **25 条**行级引用,文件存在 + 行号在范围,**0 异常**。

**结论**:漂移检测确实能抓出生成器(`repowiki` 生成的 `REPOWIKI`)和人工文档遗漏的漂移——**差异化价值已实证,不是纸面设计**。这正是 deepwiki-skill / repowiki / 通义灵码都没覆盖的缺口。报告见 [`docs/drift-report.md`](drift-report.md)。
